# prompt-codec v0.3 — persistent cache, warm model, savings telemetry

Date: 2026-07-18
Status: approved by Wayne ("Do the top three" — plain-language roadmap items 1–3)
Baseline: v0.2.0 (commit c768517), spec 2026-07-17-rust-rewrite-design.md

## Goals

Three additive features, none changing existing request semantics:

1. **Persistent rewrite cache** — accepted LLM rewrites survive proxy
   restarts and are shared with CLI one-shots, so compressed history stays
   byte-stable across restarts (protects upstream prompt-cache discounts)
   and no compression is ever paid for twice.
2. **Warm model** — the local model stays resident, making hybrid latency
   consistently ~1 s instead of ~10 s after idle unload.
3. **Savings telemetry** — cumulative token/$ savings plus upstream
   cached-token counts, visible in `/health`, persisted across restarts.

## Non-goals

- No fine-tuning infrastructure, no traffic/pair logger (future item).
- No change to compression policy, acceptance guard, or proxy forwarding
  semantics. The response byte-stream stays verbatim.
- No Anthropic-native protocol support.

## Feature 1: persistent rewrite cache

**Design:** two-tier read-through/write-through. moka (in-memory, existing)
stays the hot tier; SQLite becomes the durable tier via `rusqlite` with the
`bundled` feature (no system dependency).

- Path: `cache.path` config; default `~/Library/Caches/prompt-codec/rewrites.sqlite3`
  on macOS via `dirs::cache_dir()` + `prompt-codec/rewrites.sqlite3`
  (falls back to `./prompt-codec-cache.sqlite3` if no cache dir resolves).
  `cache.persist: true` (default) enables the tier; `false` = v0.2 behavior.
- Schema: `rewrites(key TEXT PRIMARY KEY, value TEXT NOT NULL,
  created_at INTEGER NOT NULL, last_used INTEGER NOT NULL)` plus a
  `meta(k TEXT PRIMARY KEY, v TEXT)` table (used by Feature 3 totals and a
  `schema_version` row).
- WAL journal mode (safe for proxy + concurrent CLI one-shots), busy_timeout
  250 ms — a contended lock may stall a single request up to that bound
  (rare: requires a concurrent CLI write) before degrading to memory-only;
  accepted. `rusqlite::Connection` is not `Sync`; the disk tier lives behind
  `Arc<Mutex<Connection>>` inside the already-`Clone` `RewriteCache`.
- `get`: memory hit → return. Memory miss → disk lookup; disk hit promotes
  into memory and touches `last_used` (best-effort, non-blocking semantics —
  a failed touch is ignored).
- `put`: insert into memory and disk (INSERT OR REPLACE).
- Pruning: `cache.max_disk_entries` (default 100_000). After a put, with
  probability ~1/1024 (cheap modulo on a counter — no RNG needed), delete
  oldest-by-`last_used` rows beyond the cap via
  `DELETE FROM rewrites WHERE key IN (SELECT key FROM rewrites ORDER BY
  last_used ASC LIMIT :n)` (bundled SQLite does not reliably support
  `DELETE ... LIMIT` directly).
- **Failure posture:** any SQLite error (open, read, write, prune) logs one
  `tracing::warn!` (deduplicated to once per process for open failures) and
  degrades that operation to memory-only. A broken disk cache must never
  fail or slow a request beyond the failed call itself.
- Timestamps come from `SystemTime::now()` (unix seconds).
- `/health` reports `cache_entries` (memory tier, as today) plus
  `cache_disk_entries: Option<u64>` (None when persistence is off/broken)
  and `cache_path`.
- Because keys already include model + target_ratio + content hash, config
  changes naturally miss the old entries; no invalidation logic needed.
- Blocking I/O note: rusqlite calls are synchronous. Individual point
  reads/writes on a WAL database are sub-millisecond; they run inline on the
  async path (same budget class as tokenization, accepted by the v2 spec).
  The prune statement is bounded (`LIMIT`ed delete) to keep worst case small.

## Feature 2: warm model (keep-alive pinner)

**Problem:** Ollama unloads idle models (default ~5 min); the next hybrid
call pays a ~10 s reload (measured live). Consistent ~1 s latency requires
the model to stay resident.

**Design:** config `local.keep_alive` (string, default `"60m"`, `""`
disables). The **proxy only** (never CLI one-shots), and only when
`encoder.mode` is `local`/`hybrid` (pinning a model that `rules` mode never
calls would waste RAM), runs a background tokio task:

- On startup and every 20 minutes thereafter, POST to the Ollama **native**
  API — `{base_url minus trailing /v1}/api/generate` with body
  `{"model": <local.model>, "prompt": "", "keep_alive": <keep_alive>}`.
  An empty-prompt generate is Ollama's documented model-load/pin call; it
  returns immediately after ensuring the model is resident with the given
  keep_alive.
- The pin uses a 120 s request timeout (first pin may cold-load the model)
  and runs concurrently with serving — it must not delay startup binding.
- **Non-Ollama servers** (MLX-LM, Exo): the native path 404s or refuses —
  after the first failure, log one `tracing::warn!` ("keep-alive pinning
  disabled: server does not expose Ollama's native API") and stop the task.
  Never retry-spam a server that doesn't support it.
- The OpenAI-compatible chat payload is untouched (no `keep_alive` field —
  Ollama's compat endpoint ignores it and other servers may reject it).
- `/health` includes `keep_alive: <configured value>` in the local section
  when pinning is active.

## Feature 3: savings telemetry

**Counters (per-process, atomic):** requests seen by the two compressing
endpoints, before/after token totals, saved-token total. Derived on read:
estimated USD saved (`saved_total / 1e6 × stats.usd_per_mtok_input`).

**Upstream cached-token capture:** providers report prompt-cache usage in
response bodies (`usage.prompt_tokens_details.cached_tokens`, OpenAI shape;
xAI compatible). The proxy must NOT buffer or parse response bodies on the
hot path, so:

- The forward stream is wrapped in a pass-through inspector that copies the
  **last ≤ 16 KB** of the body into a ring buffer as chunks flow through
  (bytes are forwarded unchanged, zero added latency, no full-body buffering).
- On stream end, a regex scan of the tail extracts `"cached_tokens":<n>`
  (and `"prompt_tokens":<n>` when adjacent) — works for non-streaming JSON
  (usage sits near the end) and for streaming when the client requested
  `stream_options.include_usage` (usage arrives in the final SSE chunk).
  When absent, nothing is recorded — cached-token totals are best-effort by
  design and documented as such.
- Only `/v1/chat/completions` and `/v1/completions` responses are inspected;
  catch-all traffic is never touched.

**Persistence:** lifetime totals live in the Feature-1 `meta` table
(`totals_json` row). Flushed every 32 counted requests and on graceful
shutdown (ctrl-c path already exists); loaded at proxy startup. A hard kill
may lose at most the last few requests' worth of counting — acceptable.
When `cache.persist: false`, totals are session-only.

**Surface:** `/health` gains

```json
"totals": {
  "requests": 1234,
  "before_tokens": 5678901,
  "after_tokens": 3456789,
  "saved_tokens": 2222112,
  "est_usd_saved": 6.67,
  "upstream_cached_tokens": 890123,
  "responses_with_cache_info": 1100,
  "since": "2026-07-18T17:00:00Z"
}
```

`since` is the timestamp the persisted totals row was first created (process
start when `cache.persist: false`). `totals.requests` counts only requests
that actually produced compression stats (chat with a valid messages array;
completions with a non-empty string prompt) — passthrough/malformed requests
don't count. The per-request stats log line and `x-prompt-codec-*` headers
are unchanged.

## Config additions

| Key | Default | Meaning |
|---|---|---|
| `cache.persist` | `true` | enable the SQLite tier |
| `cache.path` | platform cache dir | override DB location |
| `cache.max_disk_entries` | `100000` | disk prune threshold |
| `local.keep_alive` | `"60m"` | model residency window; `""` disables pinning |

All additive; `KNOWN_SECTIONS` and the anti-drift test grow accordingly.
Version bumps to 0.3.0.

## Error handling principles (unchanged posture)

Every new failure mode degrades to v0.2 behavior with a warn: broken disk
cache → memory-only; failed pin → cold-load latency returns; missing usage
data → counters simply don't grow. No new failure can break a request.

## Testing

- **Cache:** unit tests for the SQLite tier (put/get round-trip, disk hit
  after memory clear, prune beyond cap, corrupted-DB-file degrades with
  warning not panic); integration test proving a REBUILT `Codec` (fresh
  memory tier, same DB path via tempdir) serves a prior rewrite with zero
  LLM calls (wiremock `expect(1)` across two Codec instances).
- **Keep-alive:** wiremock native-endpoint mock asserting the pin body
  (model, empty prompt, keep_alive); a 404 mock asserting the task stops
  after one failure (no second request within a short window).
- **Telemetry:** tail-inspector unit tests (usage in non-stream JSON;
  usage in final SSE chunk; absent usage; usage string split across two
  chunks straddling the ring-buffer boundary); /health totals integration
  test (two proxied requests → requests==2, saved>0); persistence test
  (totals survive proxy app rebuild against same DB).
- All existing 92 tests keep passing untouched; the response byte-stream
  fidelity tests (SSE byte-identity, causality streaming) are the guard
  that the inspector adds no distortion.

## Acceptance criteria

1. `cargo test` green (existing + new), clippy/fmt clean.
2. Restart resilience: encode via CLI, then again in a NEW process — second
   run notes `cache_hit`, zero LLM calls, byte-identical output.
3. With the proxy idle >5 min against real Ollama, next hybrid request
   completes in ~1–2 s (no cold reload) while pinning is active.
4. `/health` shows totals growing across requests and surviving a proxy
   restart; SSE byte-identity and causality streaming tests still pass.
5. Non-Ollama upstream (wiremock): exactly one pin attempt, one warning,
   serving unaffected.
